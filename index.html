Server Side

Aphix Cloud

ERP record persistence


Core system would communicate with multiple ERPs and store that information inside CouchDB.

Direct communication is forbidden. At this point we are not considering socket type communication with the core and apps. Therefore our synchronisation and cache invalidation will be preformed by an interface layer utilized by our clients (PouchDB).

There could be potential to breakdown into smaller microservices and sync them up with bayeux protocol to retain their singularity. Providing an event system over polling.

Couch & Pouch

Single source of truth


This storage is external to all SalesRep instances and is being populated by the core system. We gain access to that information by invoking PouchDB's API to access the external data.

PouchDB is an in memory database utilizing browser storages. Our clients will utilize SQLite instead of IndexedDB/LocalStorage for persistence. Allowing for better performance as there is a smaller footprint within browsers.

Thanks to PouchDB's synchronisation we can treat PouchDB as an interface to actual source, and it would offload memory into device's memory.

UI/UX

Components


Reusability is less of a concern while system is growing, and abstracting certain elements is not always obvious. It is desireable to compose pure components which interact based on inputs and outputs.

Components which utilize state for presentational purposes or encapsulated functionality are just right. 

export default () => {
  const [counter, setCounter] = React.useState(0);

  const handleClick = React.useCallback(
    () => setCounter(oldCounter => oldCounter + 1),
    [counter],
  );

  return (
    <button type="button" onClick={handleClick}>
      {counter}
    </button>
  );
};

Views


Routing is an essential in SPAs for their conditional rendering. If we bind Redux's state on certain routed components and user is on that route. Then we get correct view according to provided query parameters.

Landed route may propagate further state changes to render the view accordingly. Then we trigger middleware to validate current state and preform further actions. 

const A = React.lazy(() => import('./A/A.component'));
const B = React.lazy(() => import('./B/B.component'));
const C = React.lazy(() => import('./C/C.component'));

export default () => (
  <Layout>
    <Switch>
      <Route path="/one" component={A} />
      <Route path="/two" component={B} />
      <Route path="/three" component={C} />
    </Switch>
  </Layout>
);

Containers


Routing is an essential in SPAs for their conditional rendering. If we bind Redux's state on certain routed components and user is on that route. Then we get correct view according to provided query parameters.

Landed route may propagate further state changes to render the view accordingly. Then we trigger middleware to validate current state and preform further actions. 

export default React.memo(({ match: { params: { id } } }) => {
  const dispatch = ReactRedux.useDispatch();

  const order = ReactRedux.useSelector(
    Orders.getOrder(id),
    ReactRedux.shallowEqual,
  );

  const dispatchProps = {
    handleConfirmOrder: () => (
      dispatch(Order.confirm(order))
    ),
    removeOrder: () => (
      dispatch(Order.remove(id))
    ),
  };

  return <Orders {...dispatchProps} />;
});
